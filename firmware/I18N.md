# openTRNTBL — i18n

Tiny vanilla-JS i18n layer for the firmware portal. No framework, no
build step, runs offline on the CHIP.

## Architecture

- **`firmware/i18n.js`** — defines `I18N` (3 languages: FR / EN / ES),
  `LOCALE` (active language), `t(key)` lookup, `setLang(lang)` switcher,
  `hydrateI18n()` re-translates the page on load and on switch.
- Translations are inlined as a JS object literal — no fetch, no network
  request. The whole layer adds ~6 KB to index.html.
- Locale detection order:
  1. `localStorage('opentrntbl-lang')` — user explicit choice
  2. `navigator.language` prefix (e.g. `es-AR` → `es`)
  3. fallback `fr` (current default product language)

## How to use

### Static text in HTML

```html
<h1 data-i18n="wifi.title">WiFi</h1>
<button data-i18n="wifi.connect">Connecter</button>
```

The inner text is the **fallback** if `i18n.js` doesn't load (graceful
degradation — page still works in French even if the script fails). On
load, `hydrateI18n()` swaps the text to the active locale.

### Attributes

```html
<input data-i18n-attr="placeholder:wifi.ssid" placeholder="SSID">
```

Multiple attrs separated by `;`:

```html
<input data-i18n-attr="placeholder:wifi.ssid;aria-label:wifi.ssid"
       placeholder="SSID" aria-label="SSID">
```

### JS-built strings

Dynamic content (status messages, generated rows, alert text) :

```js
status.textContent = t('dash.playing_on') + speakerName;
```

For DOM nodes built dynamically that need to re-translate when the user
switches language at runtime, expose a `refreshDynamicLabels()` function
in your code — `setLang()` calls it after swapping the locale.

```js
function refreshDynamicLabels() {
  // Re-render the status badge, the speaker list, etc.
  setStatus(currentState, currentSpeakerNames);
  loadDash();
}
```

## How to switch language at runtime

Call `setLang('en')`, `setLang('fr')`, or `setLang('es')`. The function:
1. Updates `LOCALE`
2. Persists the choice to `localStorage`
3. Re-translates all `[data-i18n]` and `[data-i18n-attr]` elements
4. Sets `<html lang="...">`
5. Calls `refreshDynamicLabels()` if defined

A switcher UI (3 buttons) is rendered in the Settings panel.

## Adding a new language

1. Add a key to `I18N` :
   ```js
   I18N.de = { /* same keys, German values */ };
   ```
2. Add a button in the Settings switcher in `index.html`.
3. Done. Browser detection picks up `de-*` automatically.

## Adding a new translation key

1. Add the key to all three (FR / EN / ES) blocks in `i18n.js`. Keep the
   alphabetical-ish grouping by section (`wifi.*`, `dash.*`, etc.).
2. Use `data-i18n="…"` in HTML or `t('…')` in JS.
3. If French is the default and you can't translate the others
   immediately, copy the FR string as a placeholder for EN/ES (the `t()`
   helper falls back to `I18N.fr` for any missing key).

## Known limitations

- **No pluralization** (`{count, plural, one {…} other {…}}`) — keep
  strings singular or use templating for now. Add MessageFormat layer
  later if needed.
- **No date/number formatting** — use `Intl.DateTimeFormat`,
  `Intl.NumberFormat` directly when needed.
- **No RTL support** — all 3 languages are LTR. Adding Arabic / Hebrew
  later requires `dir="rtl"` handling.

## Glossary discipline

Don't translate technical terms or brand identifiers:

- **SSID** — universal WiFi term, leave as-is in all languages
- **WiFi** — universal
- **Sonos** — brand
- **MP3** — codec name
- **RCA** — connector standard
- **IP** — universal
- Bitrate values (`128k`, `192k`, `320k`) — universal numeric

## Coverage

V1.0.0-alpha ships with ~50 keys covering :
- WiFi setup screen (full)
- Dashboard screen (full)
- Settings panel (full)
- Common dynamic labels (search, no networks, etc.)

Missing :
- Some dynamic Sonos status text mid-API-call
- Tooltips / aria-labels
- Error message specifics

These will be filled in as the firmware is migrated. The infrastructure
is in place — adding a new key is a 3-line change.
